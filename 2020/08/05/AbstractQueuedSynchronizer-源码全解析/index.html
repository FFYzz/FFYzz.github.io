AbstractQueuedSynchronizer 简称 AQS。AQS 基于 FIFO 等待队列可以实现 Blocking Locks 和 synchronizers(比如 semaphores 等)。AQS 中使用一个 volatile 修饰的 state 变量来表示当前的同步状态。一般情况下，state 为 0 表示锁未被获取，state > 0 表示锁已被获取，在可重入锁中，state 的值会 > 1。子类在实现 AbstractQueuedSynchronizer 抽象类的时候必须定义一个能够修改 state 状态的方法。基于 AQS 实现的同步器支持两种模式，一种是 exclusive 模式，另一种是 shared 模式。

锁的状态

AQS 中锁的状态由 state 变量表示。同前文所述，state 是一个 int 类型的变量，state == 0 时，表明锁未被获取。当 state > 0 时，表示锁已被获取。 state 可以大于 1 的原因是锁可以重入，重入一次， state 的值 + 1。

state 相关方法

state 相关的方法只有三个，一个是 get 方法，一个是 set 方法。因为 state 是 volatile 变量，所以其读写也是 volatile 读写。此外还有一个 cas 更新 state 的方法。

FIFO 等待队列

AQS 中的主要数据结构是一个基于 CLH(Craig, Landin, and Hagersten) 队列实现的变体 CLH 等待队列(后文以 CLH 队列代表 AQS 中使用的等待队列，但是必须得清楚 AQS 中使用的队列与规范定义的 CLH 队列还是不同的)。CLH 队列的深入理解可以阅读 T Craig 在 1993 年写的文章 “Building FIFO and priorityqueuing spin locks from atomic swap”。CLH 队列与普通队列类似，存在一个头节点指针 head 和尾节点指针 tail。CLH 队列中的元素用 AQS 中的内部抽象类 Node 表示。

CLH 队列中的节点数据结构 - Node

  • CLH 队列是一个双向队列，所以 prev 和 next 分别指向前后 Node。
  • waiter 是一个 Thread 类型的变量，绑定了 Node 对应的线程。
  • status 表示节点的状态，节点可以处于不同的状态，下面会进行分析。

Node 的 status 由三个常量来表示。

其实除了以上三个能用常量表示的状态之外，还有一个默认的状态，也就是在创建 Node 实例的时候，它的初始值为 0。

  • WAITING = 1，节点处于同步队列中等待的状态。
  • CANCELLED = 0x80000000，节点已经取消获取锁。
  • COND = 2，在条件等待队列中进行等待。

至于具体状态会进行如何处理，我们到后续的方法中再具体分析。

独占类型的节点 - ExclusiveNode

独占类型的节点 ExclusiveNode 继承自 Node 类，是一个空实现。用于对实例对象的具体类型进行判断。

共享类型的节点 - SharedNode

共享类型的节点 SharedNode 继承自 Node 类，也是一个空实现。用于对实例对象的具体类型进行判断。

条件等待队列中的节点 - ConditionNode

  • 条件等待队列不同于同步队列，它是一个单向队列，该队列并没有用到 Node 类中定义的 prev 和 next 变量，而是引入了 nextWaiter 变量，指向下一个条件等待队列中的节点。
  • ConditionNode 实现了 ForkJoinPool.ManagedBlocker 接口，实现了两个方法。

回到 CLH 队列

既然是个队列，那么就与普通队列一样，拥有队头引用指针和队尾引用指针。

CLH 的队列初始化方法，前面也提到了，只有在发生冲突之后才会初始化队列。初始化的逻辑也是非常简单，创建一个 ExclusiveNode 类型的 dummy head 节点，再将 head 和 tail 引用均指向它。这里 dummy head 节点表示只是一个标记节点，不代表任何获取锁的节点。

节点的入队列方法，仅用于从条件等待队列中唤醒后加入到同步队列中，逻辑也是非常清晰的：

  1. 最外层套了一个无限循环体。
  2. 将新加入的 node 节点的 prev 引用指向队列中原先的 tail 节点。
  3. 通过 cas 更新 tail 指向新加入的节点 node。
  4. 第 3 步的 cas 更新成功，说明入队成功，将队列原先队尾节点的 next 引用指向新加入的 node 节点。
  5. 最后判断原先的队尾节点的 status，如果 status < 0，说明被 cancelled 了，则唤醒当前加入节点 node 的绑定线程。

判断节点是否在队列中，从 tail 开始往前遍历。

判断给定线程是否在队列中，从 tail 往前遍历。

独占模式下，通知指定节点的后继节点。将后继节点的 status 设置为 status & ~WAITING，其实就是后继节点的 status 设为 0。并 unpark 后继节点绑定的线程。

共享模式下，往后遍历节点，如果满足节点是 SharedNode 且处于 WAITING status，则重置其 WAITING status 并且唤醒节点绑定的线程。

cleanQueue，如其方法名，清理队列。主要目的是为了把队列中所有已经 CANCELLED 的节点清除出去。如果 dummy head 节点后是 CANCELLED 节点，那么需要唤醒 dummy head 节点的下一个有效节点。

  1. 外层套了一个无限循环。从 tail 往前开始遍历。
  2. (p,q,s)组成一个三元组，p 指向当前节点的前驱节点，q 指向当前节点，s 指向后继节点。
  3. 退出该无限循环的唯一条件需要满足 1) 当前队列为空 或者 2) 遍历到了 dummy head 节点。
  4. 如果当前节点的 status < 0，也就是其状态为 CANCELLED 状态。则进行移除节点处理。移除当前节点 q 之后，需要判断 q 的前驱节点是否为 dummy head 节点，如果是的话需要 signalNext(p)，也就是唤醒 s 节点。
  5. 如前删除节点逻辑中的 p.casNext(q, s); 失败了，在后面做补偿处理。

hasQueuedPredecessors 方法如其方法名所示，来确定在队列中是否有比当前线程等待更久的线程。该方法可用于禁止抢占，用来实现一个公平的同步器。

  • 找到当前队列中的第一个有效的节点(非 CANCELLED 节点)，比较其绑定的线程是否是当前线程。

该方法一般用于判断锁是否被当前线程锁持有。

条件对象 - ConditionObject

前面我们提到 AQS 中有两种队列，一种是同步队列，另一种是条件等待队列。每一个 ConditionObject 对象实例都维护一个条件等待队列。一个线程对应一个 ConditionNode。ConditionNode 实现了 Condition 接口，Condition 接口的定义的 await/signal 方法族可以将其理解为对标 Object 类中的 wait/notify 方法。现在我们来看一下 ConditionObject 的设计与实现。

ConditionObject 类中只有两个成员变量 firstWaiter 以及 lastWaiter，指向条件等待队列的队头节点以及队尾节点。

等待方法 - await 系列

通过调用 await 方法，可以让当前节点进入到条件等待队列中等待。节点绑定的线程会一直阻塞，直到被 1) signal 通知或者被 2) 中断。

  1. 与其他方法类似，在执行 await 方法的入口，会先检查当前线程有没有被中断，如果被中断，则会抛出 IE。
  2. 随后释放锁，并保存释放锁之前的锁状态 state，随后准备进入阻塞状态。
  3. while 循环是阻塞等待的关键。如果当前 node 满足了条件并进入到了同步队列中等待，则会从 while 循环体中跳出。也就是说 node 等待的条件已经满足，并进入了同步队列。
  4. while 循环体中，如果等待过程中检测到被其他线程中断。且该中断操作将 如果 node 的 status 没有被更新，那么将会跳出 while 循环。
  5. while 循环体中，未检测到中断且 node 的 status 包括 COND status，则阻塞等待。
  6. while 循环体中，剩余的情况则自旋等待。
  7. 跳出 while 循环体表示 node 已经在同步队列中等待获取锁了。这时清理 node 的 status，将其设为 0。之后调用 acquire 方法去获取锁。
  8. 最后检查是否是中断取消，如果是中断取消，则需要将节点从单向队列中移除。

await 系列方法还有 boolean await(long time, TimeUnit unit) / long awaitNanos(long nanosTimeout) / void awaitUninterruptibly() / boolean awaitUntil(Date deadline) 。这几个方法的实现与 void await() 方法的实现都差不多，这里就不展开了。

通知方法 - signal 系列

signal() 与 signalAll() 的逻辑都在 doSignal(ConditionNode first, boolean all) 中。

doSignal 的逻辑也是非常清晰的，外部传入单向队列的头节点以及一个标识符。从头结点开始往后遍历，1) 将节点的 status 中的 COND 状态清除，2) 并将节点加入到同步队列中。

其他方法

enableWait 方法初始化 node 的属性。绑定当前线程，并设置 node 的 status 为 COND | WAITING。并返回当前锁的状态 state。该方式是将一个 ConditionNode 初始化为一个即将要 wait 的节点。类比 Object 类中的 wait 方法，在调用了 wait 方法之后也是需要释放锁的。因此调用 release 方法尝试释放锁，并返回释放锁之前的锁状态 state。

canReacquire 方法返回 node 节点是否可以重新获取锁。

将所有被中断取消的节点从单向队列中移除。是一个标准的单链表删除节点的操作，就不具体展开分析了。

返回当前 ConditionObject 对应的条件等待队列中是否有节点在等待。

AQS 中的独占锁

如文章开头思维导图所示，独占锁与共享锁基本上拥有对仗的方法。首先来看下独占锁相关的方法。

锁获取方法

tryAcquire 方法是一个空实现，会交给实现了 AQS 的类去具体实现。

tryAcquireNanos 方法入参中带了一个超时时间 nanosTimeout 参数。表示该获取锁的方法支持超时获取。

  1. 同样在获取之前会判断当前线程是否被中断。
  2. 首先调用 tryAcquire 方法尝试获取，如果获取成功，直接返回 true。获取失败则继续走后续的逻辑。
  3. 检查入参 nanosTimeout，如果 <= 0 的话，则直接返回获取失败。
  4. 调用公共方法 acquire 进行获取。返回结果是一个 int 值。如果返回 > 0，表示获取锁成功。如果 == 0，表示超时。如果 < 0 表示被中断。
  5. 根据 acquire 的结果构造 tryAcquireNanos 方法的返回值。

acquire 和 acquireInterruptibly 方法都是独占模式下调用的获取方法。可以看到内部逻辑都是委托给了上面分析的 tryAcquire 和 acquire 方法。

锁释放方法

tryRelease 方法也是一个空实现，需要被子类重写。

release 方法调用了 tryRelease 方法，根据 tryRelease 方法的返回值确定是否成功释放锁。若成功释放锁，则需要调用 signalNext() 方法，唤醒 dummy head 节点的后继节点。

锁获取方法

锁释放方法

锁共用方法

终于要分析 acquire 方法了。 acquire 方法可以算是 AQS 中最最重要的一个方法。但是在开始 acquire 方法之前，先看一下 cancelAcquire 方法。

cancelAcquire 方法的作用是将正在获取锁的节点取消获取这个动作。

  1. 更新当前节点的属性，并清理队列，将所有 CANCELLED 的节点从同步队列中删除。
  2. 处理中断标志。如果是中断导致的取消获取，且当前可处理 acquire 方法可处理中断，则返回中断标志。如果是中断导致的取消，但是 acquire 方法不可处理中断，则抛出中断异常。

acquire 方法基本上是 AQS 中最长的一个方法了。因此把带注释的代码贴上来了。如代码块中的注释所示，重点都在 for(;;)循环里面。

  1. 第一个 if 判断，能够满足该判断的一个前提就是 node 节点已经入队。且 node 节点的前驱节点不是 head dummy 节点。该 if 内部的逻辑主要为清理 cancelled 的节点。
  2. 第二个 if 判断，能够进入该逻辑的条件为 1) node 节点的前驱节点为 head 节点 或者 2) node 节点还未入队。这个逻辑里面会根据节点的类型调用 tryAcquire 尝试获取锁。
  3. 第三个 if 判断开始，与前面两个 if 判断没有关联。后面将是一连串的 if-else。
  4. 如果节点未初始化，则初始化节点。
  5. 如果节点未入队，则尝试入队。
  6. 如果节点从阻塞中醒来，但是pred 指向 head(也就是说还有其他线程持有锁，但是很快就要释放了)，则需要自旋等待。
  7. 如果节点的 status 未设置，则设置为 WAITING。
  8. 最后是一段阻塞的逻辑。阻塞醒来需要清除 status，且需要判断是否是被其他线程中断从阻塞中醒过来的还是其他原因。如果是被中断，则需要跳出循环，取消获取。

AQS 的使用实例

基于 AQS 的不可重入互斥锁实现

该例子提供了一个基于 AbstractQueuedSynchronizer 实现的不可重入的独占锁 Mutex 类。在类的内部定义了一个继承自 AbstractQueuedSynchronizer 抽象类的 Sync 类，用于获取锁以及释放锁。Mutex 类在内部持有一个 Sync 的实例，将所有的方法实现都委托给了内部类 Sync 来处理。

基于 AQS 的 latch 实现

该实现类似于 CountDownLatch 的功能。同样需要定义一个内部类 Sync 继承自 AbstractQueuedSynchronizer。因为 latch 的特点是非独占的，因此需要重写 AbstractQueuedSynchronizer 类中的 xxxShared 方法。
