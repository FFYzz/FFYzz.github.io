ReentrantReadWriteLock 源码全解析

发表于 2020-08-05 更新于 2020-12-13 阅读次数： Disqus：

ReentrantReadWriteLock 是基于 AQS 实现的读写锁。与 StampedLock 不同，ReentrantReadWriteLock 的写锁是支持重入的。ReentrantReadWriteLock 对外暴露的 API 主要如下图所示。ReentrantReadWriteLock 中的主要方法都在内部类中，向外暴露的都是一些锁状态监控方法，因此本文会将重点放在内部类的具体实现上。

ReentrantReadWriteLock 中的公开方法

构造方法

ReentrantReadWriteLock 中提供了两个构造方法，一个是带参数的构造方法，一个是不带参数的构造方法。传入的参数表示初始化公平锁或者非公平锁。同时会初始化 ReadLock 和 WriteLock 实例。

写锁获取方法

使用 ReentrantReadWriteLock 时可以通过 ReentrantReadWriteLock 的实例调用该方法获取到一个写锁的实例。 writerLock 在 ReentrantReadWriteLock 的构造方法中已经初始化。

读锁获取方法

同样，可以调用该方法获取到一个读锁的实例。 readerLock 在 ReentrantReadWriteLock 的构造方法中已经初始化。

Sync

内部类 Sync 继承自 AbstractQueuedSynchronizer。读写锁的主要的设计都在该抽象类中。

Sync 中定义的常量

Sync 类中定义了四个常量。在读写锁中，将 int 型的 state 拆成了高 16 位以及低 16 位。低 16 位用于表示写锁的持有个数。高 16 位表示读锁的持有个数。

  • SHARED_SHIFT，读锁的移位常量。
  • SHARED_UNIT，读锁增加的单位，因为读锁使用的是高 16 位，所以不是简单的 + 1。
  • MAX_COUNT，最大的锁个数，因为最多只能用 16 位表示，因此最大值为 (1 << SHARED_SHIFT) - 1。
  • EXCLUSIVE_MASK，独占锁个数计算时的掩码，低 16 位全为 1，高 16 位全为 0。

Sync 中的内部类

ThreadLocalHoldCounter 继承了 ThreadLocal 类，内部保存的是 HoldCounter。因为是 ThreadLocal 类型，因此每个线程都会保存一份副本。HoldCounter 是 Sync 抽象类中的内部类。HoldCounter 内部有一个 int 型的变量 count，记录了持有的读锁的个数。因此 ThreadLocalHoldCounter 与 HoldCounter 共同协作，为每个线程保存自己持有的读锁的个数。该值会缓存在 Sync 的成员变量 readHolds 中。此外，还有一个 cachedHoldCounter 成员变量，该变量保存的是最后一个获取到读锁的线程持有的读锁的个数。

Sync 中的方法

两个抽象方法

两个抽象方法留给子类具体实现，这两个抽象方法用于控制是否可抢占，也就是是否为公平锁。外部使用者可通过 ReentrantReadWriteLock 的构造方法来决定初始化公平锁或者非公平锁，内部的具体实现就由这两个抽象类来决定。

写锁的获取

tryWriteLock 方法是尝试获取写锁的方法。该方法无论是在公平锁还是非公平锁下，都允许抢占。并且 tryWriteLock 方法只支持一次性获取，获取失败不会进入队列中等待。

  1. 首先通过 getState 方法获取到当前的 state。
  2. 如果 state != 0，说明锁已经被持有。通过调用 exclusiveCount 方法得到写锁的持有数量。if 中是两个获取锁失败的情况。w == 0 表示当前存在读锁，则写锁获取失败。如果w != 0 && current != getExclusiveOwnerThread()，说明其他线程持有写锁，返回 false。如果 写锁的个数已经达到最大值，则抛出异常。
  3. 经过上面条件后，则通过 cas 尝试获取写锁。获取失败直接返回 false。
  4. 能走到最后说明获取锁成功，更新 OwnerThread 。

tryAcquire 在 ReentrantReadWriteLock 中并没有被直接调用，它的使用是在 AbstractQueuedSynchronizer 类中的 acquire 方法中。

写锁的获取方法。与 tryWriteLock 方法不同的是，tryAcquire 方法遵循公平与非公平的设定。

  1. 首先同样获取到 state。并计算得到写锁的持有个数。
  2. if (c != 0)，如果该条件满足，说明写锁或者读锁已经被持有。如果 if (w == 0 || current != getExclusiveOwnerThread()) 满足，则说明现在其他线程持有读锁，或者其他线程持有写锁。直接返回获取失败。否则，说明当前线程持有写锁，则进行重入。将 state 更新后返回 true。
  3. 如果当前锁未被持有，首先根据 writerShouldBlock() 确定是否可以抢占。如果可以抢占，则尝试 cas 获取。获取成功则更新 OwnerThread。
  4. 返回是否成功获取到锁。

写锁的释放

写锁的释放。

  1. 如果当前线程不是持有写锁的线程，则抛出异常。
  2. 更新 state 为当前的 state - releases。
  3. 如果更新后的 state == 0，表示写锁被完全释放，因此需要更新 OwnerThread 为 null。
  4. 返回值为写锁是否完全释放。

读锁的获取

tryReadLock 与 tryWriteLock 方法类似，仅尝试一次获取，获取失败不会进入等待队列中。

  1. 通过调用 getState 方法获取当前的 state。
  2. 当如果前持有写锁，且写锁的持有者不是调用线程，则返回 flse。
  3. 获取读锁的个数。如果读锁个数已到上限，则抛出异常。
  4. 排除以上所有条件之后，通过 cas 更新读锁的个数。
  5. 更新 state 成功之后还需更新其他属性。if (r == 0)，当前线程是成功获取到读锁的第一个线程，则需要将 firstReader 指向当前线程，且更新 firstReaderHoldCount 为 1。如果满足 firstReader == current，则 firstReaderHoldCount 自增。否则，更新 cachedHoldCounter 以及当前线程持有的 ThreadLocal 对象。

tryAcquireShared 方法同样未在 ReentrantReadWriteLock 中被调用，而是在 AbstractQueuedSynchronizer 类被使用到。

读锁的获取方法，能够成功获取到读锁的情况有两种，1) 线程仅持有读锁。2) 当前线程持有读锁。

  1. 如果当前的 state，如果其他线程持有写锁，则直接返回获取失败。
  2. 通过 sharedCount 方法得到当前持有的读锁的个数。如果不支持抢占且当前读锁的个数未达到最大，则通过 cas 尝试更新 state，如果更新成功，则说明获取锁成功。后面对 HoldCount 的处理与 tryReadLock 方法中的逻辑是一致的。
  3. 如果获取读锁失败，则调用 fullTryAcquireShared 方法获取锁。

fullTryAcquireShared 如其方法名所示，是 tryAcquireShared 的完整版方法。该方法与 tryAcquireShared 有相同的部分，但是更加简单。

  1. 得到 state 的值。如果持有写锁且持有写锁的线程不是当前线程，则直接返回 -1，表示获取失败。
  2. else if (readerShouldBlock())。readerShouldBlock 的实现在公平锁中和非公平锁中不同。在公平锁中，返回队列中在该线程代表的节点的前面是否有其他线程在等待。在非公平锁中，返回队列中的第一个节点是否是独占节点。如果 readerShouldBlock 返回 false。也就是说允许抢占，则直接走下面的获取锁逻辑。如果 readerShouldBlock 返回 true，表示不允许抢占，如果是第一个获取到读锁的线程，表示是重入，则直接可以走后面的逻辑获取锁。随后继续检查是否重入，如果当前线程持有的 HoldCounter 中的 count 为 0。表示未持有读锁，无法重入，需要到同步队列中等待获取锁。
  3. 如果读锁的个数已经达到最大，则抛出异常。
  4. 通过 cas 尝试更新 state。如果更新成功，则处理 firstReader 或者 HoldCounter，处理逻辑与前面的获取方法一致。

读锁的释放

读锁的释放。

  1. 如果当前线程是第一个获取到读锁的线程，则仅需要更新 firstReader 或者 firstReaderHoldCount。
  2. 否则，首先尝试使用 cachedHoldCounter，cachedHoldCounter 表示最后一个获取到读锁的线程。如果当前线程不是最后一个获取到读锁的线程，则需要通过当前线程持有的 ThreadLocal 变量 readHolds 来获取当前线程持有的 HoldCounter 。如果 count == 1，则调用 ThreadLocal 的 remove 方法，将其删除，可以避免内存泄露。如果 count <= 0，说明状态不太，抛出异常。正常情况下只要 count 自减即可。
  3. 处理 cas 更新 state。外面套了一个无限循环，直到成功更新。

NonfairSync

NonfairSync 继承了 Sync 类，重写了 Sync 类中的两个抽象方法。

writerShouldBlock 方法在 NonfairSync 中的实现直接返回 false。因为 NonfairSync 是非公平锁，所以允许抢占，而 writerShouldBlock 直接返回 false 表示不用阻塞，可以直接抢占。

readerShouldBlock 方法的返回值是 apparentlyFirstQueuedIsExclusive 方法的返回值。apparentlyFirstQueuedIsExclusive 方法返回同步队列中的第一个元素是否为独占类型的节点。

FairSync

公平锁下，writerShouldBlock 和 readerShouldBlock 方法的返回值均为 hasQueuedPredecessors 方法的返回值。hasQueuedPredecessors 方法的返回值代表同步队列中是否有其他线程在调用线程的前面在等待获取锁。
