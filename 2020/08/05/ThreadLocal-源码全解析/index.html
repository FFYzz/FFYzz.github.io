ThreadLocal 源码全解析

发表于 2020-08-05 阅读次数： Disqus：

前言

本文的 JDK 源码解读将基于 JDK14-GA 版本。

ThreadLocal 实例是一个共享对象，多线程环境下每个线程都会创建一份副本，因此初始的情况下每个线程具有相同的初始值，但是后续每个线程对 ThreadLocal 的修改是相互独立的。因为 ThreadLocal 是线程内的变量，因此不存在多线程竞争造成的问题。

类结构图

在分析 ThreadLocal 源码之前，先给出 ThreadLocal 的类结构图。

上面的类关系图基本上包括了 ThreadLocal 类的主要构件。从图中可以看出，Thread 类中持有一个类型为 ThreadLocalMap 的属性 threadLocals。每个线程都会维护自己的 threadLocals 属性，因此不会存在多线程竞争造成的数据不一致的问题。ThreadLocal 类中有一个内部类 ThreadLocalMap，这个类是实现 ThreadLocal 的核心类。可以把 ThreadLocalMap 理解为是一个 Map。同时，ThreadLocalMap 中有一个 Entry 的内部类。Entry 继承自 WeakReference，因此指向 Entry 对象的引用是一个弱引用类型。弱引用类型在 YGC 的时候会被回收。对于四种不同强度的引用类型可以查阅其他资料。

除了类结构图外，下图是一张 ThreadLocal 对象的内存分布图。

其中的虚线表示弱引用，实线表示强引用。
使用 ThreadLocal 存在两个隐患：

  1. 脏数据
  2. 内存泄露

脏数据

脏数据的出现一般出现在线程池中。在线程池中，线程会被复用，如果线程持有的 threadLocals 没有调用 remove() 方法进行清理，那么下一个获取该线程的任务如果不调用 set() 方法对 ThreadLocal 进行设置，会读取到上一个使用该线程的任务的脏数据。

解决该问题的方案则是在线程使用完 ThreadLocal 后，调用其 remove() 方法进行清理即可。

内存泄露

如上内存分布图所示，当线程在执行过程中，若栈中的 ThreadLocal 对象引用指向 null，那么堆中的 ThreadLocal 对象实例仅剩下 Entry 对象中的弱引用。因为仅有弱引用指向在发生 YGC 时会被回收，因此当发生 YGC 后，堆中的 ThreadLocal 对象会被回收，Entry 对象中的 key 即指向了 null。栈中的 ThreadLocal 对象引用之所以会被指向 null，是因为程序员意图控制使得该 ThreadLocal 不再使用，因此与之相关的 value 应该被回收。虽然 ThreadLocal 在堆上分配的内存被回收了，但是其存储的值 value 无法被回收，造成了内存泄露的问题。

解决该问题的方法与脏数据问题的处理类似，在使用完 ThreadLocal 之后调用 remove() 方法对其进行清理。

源码拆解

ThreadLocal 类的全源码注解可参看本人的中文 JDK git 仓库。

ThreadLocal 的所有属性由下表所示。

其中有两个内部类，SuppliedThreadLocal 以及 ThreadLocalMap。我们这里将重点分析 ThreadLocalMap。SuppliedThreadLocal 是 JDK8 引入的支持以 Supplier 接口进行初始化的内部类。

ThreadLocalMap

前面提到了，ThreadLocalMap 类似于 HashMap，也是一个 Map。首先先看下 ThreadLocalMap 中的属性。

由源码可见，ThreadLocalMap 中的数据结构由内部类 Entry 来表示。Entry 继承自 WeakReference。Entry 中的 k 引用类型为弱引用类型，因此当发生 YGC 情况下，Entry 中的 k 指向的 ThreadLocal 对象若仅剩下该弱引用，那么在堆中的 ThreadLocal 对象将会被回收。

此外，Entry 对象存放在 table 中，table 默认的长度为 16，并且该长度仅能为 2 的 n 次幂。当达到阈值时，table 会进行扩容，该阈值为 2/3 的 table 的长度。

size 保存的是当前 table 中实际存在的 Entry 个数。包括 key == null 的 Entry。

nextHashCode

ThreadLocal 类中提供了一个 nextHashCode 方法。前文讲到过 ThreadLocal 中使用了一个 ThreadLocalMap 用来存放值。nextHashCode 方法用于计算 Map 中 key 的 Hash 值，以计算 ThreadLocal 在 table 中存放的位置。值得注意的是，ThreadLocal 中的 Map 解决 Hash 冲突的方法并不像 HashMap 使用链地址法，而是使用 线性探测法 来解决冲突。

get

该 get 方法为 ThreadLocal 类中的实现。该方法可分为如下几部分：

  1. 获取当前线程，从获取当前线程持有的 ThreadLocalMap。
  2. 如果 ThreadLocalMap 已初始化 && 当前 ThreadLocal 作为 key 的 Entry 不为 null，则从 Map 中获取值。
  3. 如果 ThreadLocalMap 未初始化，或者已初始化但是当前 ThreadLocal 未存储值,则调用 setInitialValue 方法。

getEntry

getEntry 是内部类 ThreadLocalMap 中定义的方法。该方法可拆分为如下几个部分：

  1. 根据 key，也就是 ThreadLocal 实例计算其在 table 中的位置。计算的方法使用了比较常规的掩码计算的方式。因为 table.length 为 2 的 n 次幂，所以计算会非常快。
  2. 根据位置获取到 table 对应位置上的 Entry。如果对应位置上的 Entry 不为 null 且 key 相等(引用相等)，那么直接返回该 Entry。
  3. 否则会调用 getEntryAfterMiss 方法。

getEntryAfterMiss

一般情况下，会调用到 getEntryAfterMiss 方法的原因是 发生了哈希冲突，导致 getEntry 方法中 e.get() == key 的条件不成立。

getEntryAfterMiss 方法中的逻辑也非常清晰，直接看 while 循环中的三个判断条件即可：

  1. 如果满足 k == key，那么说明找到了正确的 Entry。
  2. 如果满足 k == null，说明该位置上的 ThreadLocal 被回收了，但是 key 对应的 value 还未被回收，因此可能会造成前面提到的内存泄露，所以需要调用 expungeStaleEntry 方法。
  3. 不满足以上两个条件就往后找。
  4. 若都找不到，则返回 null。

expungeStaleEntry 方法将放在后面在分析，这个方法是用于解决内存泄露问题的核心方法。

setInitialValue

setInitialValue 可以拆成一下几部分：

  1. 一进入 setInitialValue 方法，首先会调用 initialValue 方法。initialValue 在 ThreadLocal 方法中的实现直接返回 null。如果有初始化的需要，那么子类在继承 ThreadLocal 的时候需要重写 initialValue 方法。
  2. 同样获取当前线程的绑定的 ThreadLocalMap，若 map 不为 null，则将初始值设置进去。
  3. 如果 map 为 null，则初始化 map，同样将 默认值设置进去。
  4. 如果当前的 ThreadLocal 实现是 TerminatingThreadLocal 类或者其子类。那么会向 REGISTRY 中进行注册。同样在 remove 的时候户进行反注册。
  5. 返回初始值。

set

首先还是看下 ThreadLocal 中的 set 方法。

同样将其拆解：

  1. 获取当前线程以及其绑定的 ThreadLocalMap。
  2. 若 map 未初始化，则创建 map。
  3. 若 map 已初始化，则将值设置进 map。

此时调用到了 ThreadLocalMap 中的 set 方法。

ThreadLocalMap#set

该方法将 Entry 设置仅 table 中。重点同样在 for 循环中：

  1. for 循环的退出条件的 e == null。若通过该条件退出循环，表明需要在 table 中放入一个新的 Entry。并且放入之后需要调用 cleanSomeSlots 方法。该方法同样放到后面拆解 expungeStaleEntry 方法时再分析。
  2. 进入 for 循环，表明该位置上的 Entry 不为 null。
  3. 当条件 k == key 满足时，相当于覆盖旧值。
  4. 当条件 k == null 满足时，说明某一个 ThreadLocal 被回收了，可能会发生内存泄露。调用 replaceStaleEntry 方法处理。同样放到后面再分析。

remove

同样先看下 ThreadLocal 类中的 remove 方法。

该方法简洁明了，获取当前线程之后，如果当前线程持有的 ThreadLocalMap 不为 null，则委托给 ThreadLocalMap 中的 remove 方法进行清理。

ThreadLocalMap#remove

重点同样关注 for 循环：

  1. for 循环的退出条件为 e == null，若在此条件下退出，表明该 ThreadLocal 早已被移出 ThreadLocalMap。
  2. 若找到对应的 key，将其指针指向 null，value 由后续的 expungeStaleEntry 方法进行处理。

expungeStaleEntry

本小节将分析以下三个方法：expungeStaleEntry / expungeStaleEntries / replaceStaleEntry / cleanSomeSlots。

expungeStaleEntry

首先来看 expungeStaleEntry 方法。

该方法的功能如其方法名所示，消除 stale Entry。对于 Stale Entry 的定义是这样的。如果一个 Entry 的 key == null，但是 value 并不等于 null。那么将其称为是一个 stale Entry。

  1. 首先将要 expunge 的位置的 value 置为 null，并将 table 相应位置也置为 null。这里仅仅完成了最基本的处理。处理还在后头。
  2. for 循环开始处理从 staleSlot 到 tab[i] == null 中间可能遇到的 stale entry。为什么要这样处理呢？因为在 staleSlot 位置可能发生哈希冲突，因此原先计算出来位置在 staleSlot 上的 Entry 往后放了，所以在将该位置处理完之后，需要把后续计算出来的位置在 staleSlot 上的 Entry 移动。
  3. 如果 k == null 条件成立，说明该位置上的 Entry 是一个 stale Entry。需要消除。
  4. 否则，如果再满足 h != i 条件，该条件满足说明该 Entry 存放的位置是因为发生了哈希冲突之后根据线性探测法后移才出现在这里。因此需要计算出来它应该存放的位置，然后往后找到一个可以存放的位置。
  5. 返回最后处理到的位置。

expungeStaleEntries

与 expungeStaleEntry 方法名相似的还有一个方法 expungeStaleEntries。

该方法进行全局的处理。在 rehash 方法中调用。

replaceStaleEntry

replaceStaleEntry 方法在 ThreadLocalMap#set 方法中被调用。当时的调用条件时这样的。

也就是调用 set 方法时，当前位置的 Entry 为 stale Entry，则会执行该方法。

  1. 首先看到第一个 for 循环。从传入的 staleSlot 位置往前找，尝试找到一个 stale Entry。若找到，则更新 slotToExpunge 为找到的位置的 index。否则 slotToExpunge 依然为传入的 staleSlot。
  2. 第二个 for 循环。从传入的 staleSlot 位置开始往后找，直到 table[i] == null。在找的过程中，如果满足条件 k == key。那么就设置其 value 值。把原来传入的 staleSlot 位置设置为非 stale Entry。把当前遍历到的位置 i 设置为 stale Entry。如果之前往前找没有找到 stale Enrty，则将 slotToExpunge 更新为当前位置 i。随后调用 expungeStaleEntry 以及 cleanSomeSlots 进行清理。
  3. 同样还在第二个 for 循环中。若 没有找到 k == key，表明又遇到了 stale Entry，且往前找的时候没有找到 stale Entry。则更新 slotToExpunge 为当前的 i。
  4. 跳出第二个 for 循环表明没有找到传入的 key 与 table 中的 Entries 中的 key 匹配。则在 staleSlot 位置进行更新，直接将其指向一个新的 Entry。
  5. 最后再判断上述过程是否有找到其他的 stale Entry，若有，则调用 expungeStaleEntry 以及 cleanSomeSlots 进行清理。

cleanSomeSlots

该方法是执行效率与内存管理折衷之后的结果。
首先由两种类型的方法会调用该方法，这两个入口传入的参数意义不同：

  1. set 方法调用，传入的 n 为当前 table 中元素的个数。
  2. replaceStaleEntry 方法调用，传入的 n 为当前 table 的 length。

重点仍然在循环体中：

  1. 循环最多次数为 log2(n) 次。
  2. do 循环中首先调用 nextIndex 方法是因为 expungeStaleEntry 方法返回的 index 为最后处理过的索引，所以从下一个位置开始找。
  3. e != null && e.get() == null 该条件为寻找 stale Entry。如果找到，调用 expungeStaleEntry 进行处理。
  4. 返回的结果为是否有过处理 stale Entry。

rehash

rehash 方法的调用条件为 ThreadLocalMap#set 中的：

回到 rehash 方法，可以看到：

  1. 全局处理 StaleEntries。
  2. 判断 size 是否超过阈值，若超过阈值则进行 resize。

resize

resize 方法挺长的，但是思路非常清晰：

  1. 新建一个 2 倍长度的 table。
  2. 看到 for 循环，如果满足条件 k == null，那么说明该 Entry 为 stale Entry，不必移到新的 table 中，因此只需将 value 指向 null，帮助 GC。
  3. 否则，计算新的位置，将 Entry 放入正确的位置中，中间使用了线性探测法解决哈希冲突。
  4. 设置新的阈值。
  5. 设置 size。
  6. 指向新的 table。

Best Practice

{@code ThreadLocal} instances are typically private static fields in classes that wish to associate state with a thread (e.g., a user ID or Transaction ID).

  • 为了解决脏数据以及内存泄露问题，在每一次使用完 ThreadLocal 后都需要手动调用 remove 方法。
  • 根据源码中的注释提示，ThreadLocal 实例一般设置成 private static。
