Condition接口 的完整源码注释可以参看本人 Github 中的 JDK14 仓库

Condition 接口的存在是为了对标 Object 类中的 wait 和 notify 方法。因此可以将 Condition 接口作为使用更加灵活的线程间通信工具。

API

await系列

线程释放锁并进入阻塞状态。只有如下三种方式能够从阻塞状态中唤醒过来。

  1. 其他线程调用 signal/signalAll 方法。
  2. 被其他线程中断。
  3. spurious wakeup 虚假唤醒。

spurious wakeup：spurious wakeup 现象指的是在没有其他线程调用 signal 方法的情况下，await 的线程会被唤醒，这种现象会出现在 POSIX Threads 和 Windows multithreading APIs 中。所以在使用 await 方法的时候，最好在外部使用一个 while 的判断条件。比如下面生产者消费者 demo 中的例子。

以上三个方法都是传入了时间参数的 await 方法，有如下几种方式能从阻塞状态中唤醒。

  1. 被其他线程调用 signal/signalAll 唤醒。
  2. 被其他线程中断。
  3. 达到设置的超时时间。
  4. spurious wakeup 虚假唤醒。

该 await 方法不能响应中断，因此只有如下两种方式能够从阻塞状态中唤醒。

  1. 其他线程调用 signal/signalAll 方法。
  2. spurious wakeup 虚假唤醒。

signal系列

signal 方法唤醒条件等待队列中的一个阻塞线程。

signalAll 方法唤醒条件等待队列中的所有线程。
