ReentrantLock 具有和 synchronized 关键字相同的行为和语义，包括同步控制、可重入等。但是 ReentrantLock 在使用上更加灵活，在功能上更加全面(比如支持公平锁与非公平锁)。ReentrantLock 的实现的基于 AbstractQueuedSynchronizer 实现的。ReentrantLock 实现了 Lock 接口，内部类 Sync 继承自 AbstractQueuedSynchronizer，类关系图如下图所示。

ReentrantLock 类中的 API 由下图所示，基本上所有的方法实现都委托给了持有的 Sync 实例实现。因此， ReentrantLock 的源码解读主要是解读 Sync 内部类的具体实现。

ReentrantLock 类中的方法

构造方法

因为 ReentrantLock 支持公平锁与非公平锁的实现。因此在 ReentrantLock 类的构造方法中，支持指定生成 公平锁实例或者非公平锁实例。

无参默认构造函数，默认初始化一个非公平锁的实例。

有参构造函数，传入一个 boolean 型的参数。传入 true 表示初始化一个公平锁实例，传入 false 表示初始化一个非公平锁实例。

其他 API 都是委托给了 Sync 内部类实现，因此留到 Sync 内部类分析。

Sync 内部类

Sync 是一个内部抽象类，抽象类的作用类似于模板方法。在 Sync 类中确定了一些公共行为，在其实现类中可以实现重写实现自己特定的行为。

加锁方法

Sync 内部类中定义了四个加锁的方法。分别是 lock/lockInterruptibly/tryLock/tryLockNanos 。其中 initialTryLock 是一个抽象方法。具体的实现交给 Sync 的子类(FairSync 以及 NonfairSync)实现。这里提前剧透一下 initialTryLock 的作用，其实 initialTryLock 是一个快速尝试获取锁的方法，只有当 1) 锁未被其他线程持有 或者 2) 锁被当前线程持有，请求重入。上述两种情况时才会返回 true。

lock 方法:

  1. 如果 initialTryLock 获取锁失败，则调用 acquire 方法去获取锁。acquire 的实现也在 AbstractQueuedSynchronizer 中。

lockInterruptibly 方法:

  1. 在执行获取锁的逻辑之前会先检查当前线程是否被中断，如果被中断，则直接抛出 IE。
  2. 与 Lock 方法类似，先通过快速获取锁方法 initialTryLock 尝试获取锁，如果获取失败，调用 AbstractQueuedSynchronizer 中实现的 acquireInterruptibly 获取锁。

tryLock 方法:

  1. tryLock 方法有个需要注意的点就是它是一个非公平的方法。无论具体的实现是 FairSync 或者是 NonfairSync，使用该方法获取锁时都能够抢占。代码的第一个 if 判断就是执行抢占获取的逻辑。
  2. 第二个 else if 则是处理重入的情况。
  3. tryLock 方法只有获取成功或者获取失败两种结果，不会进入到等待队列中去等待获取锁。

tryLockNanos 方法:

  1. 执行获取锁逻辑之前先检查中断。
  2. 调用 initialTryLock 尝试一次快速获取。获取成功直接返回，获取失败执行后续的 tryAcquireNanos 逻辑。
  3. tryAcquireNanos 的实现在 AbstractQueuedSynchronizer 中。tryAcquireNanos 的逻辑也是执行一次快速获取锁。获取失败走 AbstractQueuedSynchronizer 中的 acquire 逻辑。不过该获取锁方法是带超时取消的。

释放锁方法

Sync 内部类中提供了一个默认的释放锁方法实现。

  1. 首先判断当前线程是否是持有锁的线程，如果不是则抛出 IE。
  2. 如果释放锁之后的 state == 0，需要将锁的持有线程置为 null。
  3. 返回值为锁是否被完全释放。

其他方法

Sync 抽象内部类中的其他方法的实现就比较简单了。

getHoldCount 方法:

  • 先判断调用该方法的线程是否为持有锁的线程。如果是的话则返回当前持有锁的个数。否则返回 0。

getOwner 方法:

  • 返回持有锁的线程，如果锁未被持有，则返回 null。

isHeldExclusively 方法:

  • 返回调用线程是否是持有锁的线程。

isLocked 方法:

  • 返回锁是否被持有。

newCondition 方法:

  • 返回一个 ConditionObject 实例。AbstractQueuedSynchronizer 源码解析中提到过，每个 ConditionObject 实例都对应一个 condition queue。

FairSync 内部类

FairSync 继承自 Sync 类。ReentrantLock 类中持有的 Sync 的具体类型要么是 FairSync 类型，要么是 NonfairSync 类型。大部分的方法都在 Sync 抽象类中实现了。

initialTryLock 方法:

  1. 该方法在前面已经简单提到过了。是一个快速获取锁的方法。
  2. 先判断当前锁是否被持有，如果未被持有，先检查同步队列中是否有节点在等待获取锁(hasQueuedThreads 方法)，如果有则直接返回获取失败。这里的实现与 NonfairSync 不同之处在于这里检查了 hasQueuedThreads。因为 FairSync 是公平锁，不允许抢占，所以即使当前锁未被持有，也要检查同步队列中是否有线程在等待获取锁。
  3. 若同步队列中没有节点在等待获取锁，则尝试将 state 更新为 1。如果更新成功，则表明成功获取到锁。
  4. 如果锁已经被持有，则判断是否是在重入，如果是重入，则更新 state，并返回成功。

tryAcquire 方法:

  1. tryAcquire 方法定义在 AbstractQueuedSynchronizer 中。在 AbstractQueuedSynchronizer 类中的定义是一个空实现。
  2. tryAcquire 也是一个“不成功，便成仁”的方法。如果能够获取成功，则返回 true。如果不能获取成功，则返回 false，不会进入到同步队列中等待。
  3. 首先检查锁是否被持有，如果锁被持有，则直接返回 false。
  4. hasQueuedPredecessors 方法返回是否有比调用线程等待时间更长的线程。该方法可用于实现禁止抢占。这里使用该方法也是用于实现禁止抢占。
  5. 如果满足 1) 锁未被持有 2) 当前线程是等待队列中排在最前面的等待线程 或者 等待队列中没有等待线程。则通过 cas 更新 state，如果更新成功，则表示获取锁成功。

NonfairSync 内部类

NonfairSync 类与 FairSync 内部类类似，只不过 NonfairSync 是非公平锁的实现，允许抢占。同样来看下它的两个方法。

initialTryLock 方法:

  1. 因为非公平锁允许抢占，所以 NonfairSync 中的 initialTryLock 方法与 FairSync 中的 initialTryLock 方法不同之处在于不用检查同步队列。直接可以通过 cas 尝试更新 state。如果更新成功，则获取锁成功。
  2. 重入锁的处理逻辑与 FairSync 中的 initialTryLock 一致。

tryAcquire 方法:

  1. 非公平锁中的 tryAcquire 方法与 公平锁的 tryAcquire 方法的区别之处在于 if 判断中是否需要检查同步队列的状态。
  2. 其余逻辑是一致的。

Best Practice

使用 ReentrantLock 的最佳实践就是在调用 lock 方法之后一定需要在 finally 块中 unlock。
