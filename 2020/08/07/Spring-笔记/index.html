Srping 对事务的抽象

提供一致的事务模型。无论是使用 JDBC/MyBatis/Hinernate/Datasource/JTA，都是一致的模型。

主要定义了 PlatformTransactionManager 接口，接口中提供了基础的事物操作定义。实现该接口的类有 DataSourceTransactionManager/HibernateTransactionManager/JpaTransactionManager 等等。
TransactionDefinition 中定义了事物的一些特性。定义了事务的传播行为，隔离级别。

编程式事务

推荐通过 TransactionTemplate 来控制事务。

声明式事务

基于 AOP Proxy。

开启注解的方式：

  • @EnableTransactionManagement - 注解
  • tx:annotation-driven/ - xml

开启注解的配置属性：

  1. proxyTargetClass: true/false，可以指定使用 JDK Proxy 或者 CGLIB 增强类。
  2. mode: true/false，是否使用 AspectJ。
  3. order: 事务 AOP 拦截的优先级，默认最低优先级，那么其他 AOP 也会被包在事务里面。

@Transactional 注解可以使用在 类上或者方法上。有一些属性：

  • transactionManager
  • propagation
  • isolation
  • timeout
  • readOnly
  • 控制怎样才会回滚，rollbackFor 属性

Soring 对 JDBC 异常的抽象

会将 JDBC 常见的异常转换成 DataAccessException 抽象异常。抽象异常有很多具体的异常实现类。

如何做到？通过 SQLErrorCodeSQLExceptionTranslator 搜集。Spring 中有一个 sql-error-codes.xml 文件中定义了各个不同的数据库的 error code，并将这些 code 映射到不同的具体的继承自 SQLErrorCodeSQLExceptionTranslator 的错误实现类。每个数据库 error code 都定义了一个 Bean。

可以通过在 classpath 下自己定义 sql-error-codes.xml 来处理一些新的 error code 对应的异常。会覆盖官方的默认定义文件。

可以通过实现具体的 error code 实现类来自定义处理异常。

第五章

Project Reactor

Reactive Programming: 响应式编程。

RXJava。

Project Reactor 是 Call back 编程风格的优化版本。

核心概念

  • Operator - Publisher / Subscriber
    • Nothing Happens Until You Subscribe()
    • Flux [0,..,N] - onNext() / onComplete() / onError(): 0 - N 个元素，在每个元素遍历之后会调用 onNext 方法，全部遍历之后会调用 onComplete 方法，遇到错误之后会调用 onError 方法。
    • Mono[0,…,1] - onNext() / onComplete() / onError(): 0 或 1 个元素。
  • Backpressure
    • Subscription
    • onRequest() / onCancel() / onDispose(): publish 很快的时候 subscriber 可以通过 onRequest 控制 subscribe 的个数等等
  • 线程调度
    • immediate(): 使用当前线程处理 / single():复用某一个线程 / newSingle(): 新启一个线程
    • elastic(): 在线程池缓存中操作 / parallel(): 固定数量的线程池，与 CPU 核数一致 / newParallel(): 使用指定的线程池
  • 错误处理
    • onError: 类似于 try catch / onErrorReturn: 遇到错误后返回指定值 / onErrorResume: 遇到错误后执行某段 lambda 表达式
    • doOnError / doFinally: 类似于 finally

在没有 subscribe 之前一致操作都不会发生。

经典的那张图。

第八章

设计好的 RESTfulWebService

如何实现 Restful Web Service:

  1. 识别资源
    1. 找到领域名词
      1. 能用 CRUD 操作的名词
    2. 将资源组织为集合
    3. 将资源合并为复合资源
    4. 计算或处理函数
  2. 选择合适的资源粒度
    1. 站在服务端的角度考虑
      1. 网络效率
      2. 表述的多少
      3. 客户端的易用程度
    2. 站在客户端的角度考虑:
      1. 可缓存性
      2. 修改频率
      3. 可变性
  3. 设计 URI
    1. 使用域及子域资源进行合理的分组或划分
    2. 在 URI 的路径部分使用斜杠分隔符(/)来表示资源的层次关系
    3. 在 URI 的路径部分使用逗号(,)和分号(;)来表示非层次元素
    4. 使用连字符(-)和下划线(_)来改善路径中名称的可读性
    5. 在 URI 的查询部分使用“与”符号(&)来分隔参数
    6. 在 URI 中避免出现文件扩展名(.php等等)
  4. 选择合适的 HTTP 方法和返回码
    1. GET | 安全且幂等 | 用于信息获取
    2. POST | 不安全且不幂等 | 创建更新或一次性对多个资源进行修改
    3. DELETE | 不安全但幂等 | 删除资源
    4. PUT | 不安全但幂等 | 更新或者完全替换一个资源
    5. HEAD | 安全且幂等 | 获取与 GET 一样的 HTTP 头信息，但是没有响应体
    6. OPTIONS | 安全且幂等 | 获取资源支持的 HTTP 方法列表
    7. TRACE | 安全且幂等 | 让服务器返回其受到的 HTTP 头
  5. 设计资源的表述
    1. Json
      1. MappingJackson2HttpMessageConverter
      2. GsonHttpMessageConverter
      3. JsonbHttpMessageConverter
    2. XML
      1. MappingJackson2XmlHttpMessageConverter
      2. Jaxb2RpptElementHttpMessageConverter
    3. HTML
    4. ProtoBuf
      1. ProtobufHttpMessageConverter
