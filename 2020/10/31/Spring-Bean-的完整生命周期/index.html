Spring Bean 的完整生命周期

发表于 2020-10-31 更新于 2020-11-25 阅读次数： Disqus：

Spring 中涉及到两个比较重要的生命周期的管理，一个是 Bean 的生命周期，另一个是 ApplicationContext 的生命周期。本文将结合源码，对 Spring Bean 的全生命周期进行整理。

本文整理基于的 Spring Framework 版本为 5.2.8.RELEASE。

前言

Bean 在 Spring 中的底层数据结构表示形式是通过 BeanDefinition 来存储的。并且，Bean 的定义常见的可以有以下三种方式:

  1. 在 XML 配置中声明 bean 标签。
  2. 通过注解方式，在配置类中以 @Bean 的方式进行声明。
  3. 通过原生 API 的方式，BeanDefinitionBuilder 来创建 Beandefinition 随后注册 Beandefinition 以及最后获取到 Bean。

当然 Spring 中也允许通过 properties 文件定义 Bean，但是用的很少。

首先，给出一张差不多都见过的 Bean 生命周期图：(建议与下面分步流程解析配合使用)

Bean 元数据 BeanDefinition 的配置加载与注册

从前文可知，Bean 的定义可以有三种方式。

BeanDefinition 的加载与注册过程根据不同的资源类型有不同的加载方式，下面以 XML 资源类型的加载为例。

  1. AbstractBeanDefinitionReader#loadBeanDefinitions 方法是加载 BeanDefinition 的入口。该方法由多个重载实现，可以传入 String… 或者 Resource… 类型。
  2. 如果传入的是 String 类型的路径，则根据传入的值，配合 PathMatchingResourcePatternResolver 处理传入路径之后，构造出一个 Resource 实例。
  3. 得到 Resource 之后，调回到 AbstractBeanDefinitionReader#loadBeanDefinitions(Resource) 方法进行加载。
  4. 因为当前分析的是 XML 类型的资源，因此会调到 XmlBeanDefinitionReader#loadBeanDefinitions 方法。
  5. XmlBeanDefinitionReader#loadBeanDefinitions 会调用底层的 InputStream，读取资源文件数据。并将读取到的 InputStream 封装成 InputSource。
  6. 随后调用 XmlBeanDefinitionReader#doLoadBeanDefinitions 方法，InputSource 被解析成 Document 对象，随后对 Document 对象进行解析。
  7. 通过 BeanDefinitionDocumentReader#registerBeanDefinitions 方法对 BeanDefinition 进行注册。
  8. 注册的过程中会调用到 DefaultBeanDefinitionDocumentReader#parseBeanDefinitions 方法，进行解析。
  9. 默认可以解析四种类型的标签，import / alias / bean / beans(内部嵌套)。
  10. 解析 bean 标签的时候，DefaultBeanDefinitionDocumentReader#processBeanDefinition 会调用到 BeanDefinitionReaderUtils#registerBeanDefinition 进行注册。最终会将 <beanName, BeanDefinition> 存到 DefaultListableBeanFactory 的 beanDefinitionMap 中，把 beanName 存放到 beanDefinitionNames 列表中。列表中存放 beanName 用于维护 BeanDefinition 的顺序。
  11. 同时也在这 BeanDefinitionReaderUtils#registerBeanDefinition 中对 Bean 的别名进行处理。
  12. 可以通过继承 EmptyReaderEventListener 重写其中的方法来监听注册完成事件。

Bean Class 的加载

从前文的 BeanDefinition 配置加载与注册可以得知，仅仅是将 beanName 与 对应的 BeanDefinition 放入到 Map 中进行保存，并没有进行实例化，并且 RootBeanDefinition 中的 beanClass 也还是为 String 类型的字符串，并没有解析为目标 Bean 的 Class 类型。在 AbstractAutowireCapableBeanFactory#createBean 方法中，就会调用到 resolveBeanClass，最终通过 Class.forName API 解析得到 Bean 的 Class 类型。

Bean 实例化前的 postProcessor 回调

得到 Bean 的 Class 类型之后，同样在 AbstractAutowireCapableBeanFactory#createBean 方法中，会调用 resolveBeforeInstantiation 方法。

如果当前应用上下文注册了 InstantiationAwareBeanPostProcessor 接口，重写了 postProcessBeforeInstantiation 方法，则在这里可以对 Bean Class 进行处理，比如返回一个代理的 Bean。

实例化 Bean

AbstractAutowireCapableBeanFactory#doCreateBean 正式进入到创建 Bean 的过程中。该方法会调用 createBeanInstance 方法进行实例化 Bean。

instantiateBean 方法根据实例化策略来进行实例化。Spring 中提供了两种实例化策略，一种是基于 Java 原生反射的实现 SimpleInstantiationStrategy。另一种基于 CGLib 字节码增强的 CglibSubclassingInstantiationStrategy 策略。不同策略底层在实例化时有不同的行为。

实例化之后的行为还包括 MergedBeanDefinitionPostProcessor 动作，允许 Bean 在实例化之后有一个后置处理动作，对 Bean 进行配置。

Bean 实例化后的 postProcessor 回调

AbstractAutowireCapableBeanFactory#populateBean 方法的调用意味着进入到了 Bean 实例属性填充的阶段。如果当前应用上下文存在 InstantiationAwareBeanPostProcessor ，则调用其 postProcessAfterInstantiation 方法进行实例化之后的后置处理，也可以认为是属性填充前的处理。

Bean 属性的填充

属性在 BeanDefinition 中保存在 PropertyValues 中。因为 InstantiationAwareBeanPostProcessor 接口不仅仅定义了 postProcessBeforeInstantiation 和 postProcessAfterInstantiation 方法，还定义了 postProcessProperties 方法。因此如果存在 InstantiationAwareBeanPostProcessor 处理器，则可以调用 postProcessPropertyValues 方法，对属性的设置进行拦截。

随后调用 applyPropertyValues 方法对属性进行设置。

Aware 接口回调

AbstractAutowireCapableBeanFactory#initializeBean 方法将对 Bean 进行初始化。初始化之前会对 Aware 接口进行回调。如果 Bean 实现了以下 Aware 接口，则会进行回调。
回调的顺序是 BeanNameAware -> BeanClassLoaderAware -> BeanFactoryAware。

Bean 初始化前的处理

AbstractAutowireCapableBeanFactory#initializeBean 方法中会调用 applyBeanPostProcessorsBeforeInitialization 方法，该方法中会取出所有的 getBeanPostProcessors，并调用其 postProcessBeforeInitialization 方法。

Bean 的初始化方法

执行完 postProcessBeforeInitialization 后，紧接着调用的是 invokeInitMethods 方法。该方法将会调用 Bean 的实例化方法。此时会调用我们自己定义的 invokeInitMethods 方法。如果实现了 InitializingBean 接口，则会调用 afterPropertiesSet 方法。如果有自定义 initMethod 方法，则调用自定义的 initMethod。
其实还有一个类似的 @PostConstruct 注解标注的方法，这个标注方法在 applyBeanPostProcessorsBeforeInitialization 中被执行。严格来说是初始化前方法，而不是初始化方法。因此 @PostConstruct，InitializingBean 接口，自定义 initMethod 方法是有执行顺序的。

Bean 初始化后的处理

AbstractAutowireCapableBeanFactory#applyBeanPostProcessorsAfterInitialization 方法时 Bean 初始化后的处理。同样，如果定义了 BeanPostProcessor，则会执行初始化后的逻辑。

初始化后逻辑执行完之后，还会尝试注册 Bean 的销毁回调.

该处会根据是否实现 DisposableBean 接口或者自定义 destroy 方法来确定是否提供注册销毁回调方法。

Bean 销毁前的处理

BeanFactory 中需要手动调用 destroy，在 ApplicationContext 的 close 方法中。会执行到 DefaultSingletonBeanRegistry#destroyBean 方法，在该方法的后续调用中会调用到 postProcessBeforeDestruction。这段逻辑中可以执行 Bean 销毁前的回调逻辑。

Bean 的销毁

Bean 的销毁，如果 Bean 实现了 DisposableBean 接口，则会调用重写了 DisposableBean 的 destroy 方法。如果有自定义销毁方法的话，则会执行自定义的销毁方法。
对应于初始化情况，销毁也有一个 @PreDestroy 注解标注的方法。使用该注解标注的方法会在 postProcessBeforeDestruction 中执行，所以严格上来说是销毁前的逻辑。
因此以上三个方法也是有顺序的，即 @PreDestroy -> DisposableBean 接口 -> 自定义销毁方法。
